01 / Research Agent

带引用的研究代理

面向行业调研、竞品分析和技术选型的浏览器 agent，占位案例可替换为真实项目链接。

  • Problem手动检索耗时，证据来源分散，结论难以复查。
  • Behavior规划查询、浏览网页、提取证据、合并摘要并保留引用。
  • ToolsBrowser, Search, OpenAI, Markdown report.
  • Evaluation检查引用覆盖率、事实一致性、重复来源和不可证实结论。
  • ImpactTemplate: reduce manual research time by X% after real benchmark.
02 / Ops Workflow Agent

运营工作流代理

面向重复运营流程的 task agent，强调权限、确认和失败恢复，而不是只做聊天入口。

  • Problem跨表格、邮件和内部系统同步信息，人工容易漏项。
  • Behavior读取输入、归类任务、识别异常、生成操作建议。
  • ToolsMCP, Sheets, Email, internal API, structured logs.
  • Evaluation用任务完成率、误报率、人工确认次数衡量可靠性。
  • ImpactTemplate: save X hours/week for recurring workflow after pilot.
03 / Code Review Agent

代码审查代理

面向工程团队的 PR review agent，重点是风险定位、可操作反馈和测试信号整合。

  • Problem代码审查需要快速发现行为回归、遗漏测试和安全风险。
  • Behavior读取 diff、定位风险、生成审查意见并引用文件位置。
  • ToolsGitHub API, static analysis, tests, LLM review rubric.
  • Evaluation用真实 PR 回放评估命中率、噪声率和可执行性。
  • ImpactTemplate: improve review coverage on high-risk changes by X%.

How I Build Agents

展示的不只是会调 API，而是能把 agent 设计成可测试、可观测、可上线的产品系统。
01 Define the job and failure modes 先写清任务边界、成功标准、不可做事项和需要人工确认的动作。
02 Design tools as contracts 为工具调用设计输入输出、权限范围、错误处理和审计日志。
03 Evaluate before scaling 用回放数据、评估集和人工 review 校准 agent，而不是只看 demo 效果。

Build agents like systems, not magic tricks.

我关注 agent 的真实工程问题：任务边界是否清晰，工具权限是否可控，失败是否能恢复，输出是否可验证，以及系统是否能被持续迭代。

01先定义任务和失败模式，再决定 agent 架构。
02可观测性、评估和日志不是上线后的补丁，而是设计的一部分。
03权限、安全、人工确认和回滚路径必须内建。
